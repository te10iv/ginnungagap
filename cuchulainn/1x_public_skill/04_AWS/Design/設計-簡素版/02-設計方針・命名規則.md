2. 設計方針・命名規則
2-1. AWSリソースの命名規則

基本形式：

<system>-<env>-<role>-<seq>
例：myapp-prod-web-01


ポイント：

環境(dev/stg/prod)必須

人が見て役割が分かる名前

変更に弱い情報は入れない（AZ名など）

2-2. タグ設計

必須タグ例：

Name

env

system

owner

cost-center

backup=true/false

タグは以下に効く：

料金配布

IAM条件

CloudWatchフィルタ

IaCの動的参照

運用自動化のキー

2-3. IaC設計思想（DRY / module化）

同じコードを繰り返さない（DRY）

module で環境差分を変数に切り出す

「dev / stg / prod の3環境をコードだけで変えられる」状態にする

2-4. Git運用ルール

推奨：

main（production状態）

develop（検証）

feature/xxx（個別作業）

IaCのレビューは必須：

fmt

validate

plan diff をレビューしてから apply

2-5. 環境構成（dev / stg / prod）

dev：自由に試す

stg：本番相当のテスト

prod：厳格・制限あり

AWSでは「環境別アカウント分離」が標準
