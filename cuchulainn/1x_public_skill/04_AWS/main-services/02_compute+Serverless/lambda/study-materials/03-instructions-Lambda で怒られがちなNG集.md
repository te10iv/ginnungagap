Lambda で怒られがちなNG集

あなたは AWS Lambda を実務で運用・開発してきたプロです。
来月から Lambda を使う案件に入る初学者が
現場で怒られないための
**Lambda で怒られがちなNG集**を Markdown で作成してください。

## 目的
- 「やりがちミス」を事前に知って事故を防ぐ
- レビューや障害対応で指摘されやすいポイントを潰す
- NG → なぜダメ → どう直す をワンセットで理解する

## 出力形式（必須）
- Markdown
- 各項目は次のテンプレで書く

## NG: （短いタイトル）
- 何がNGか：
- なぜ怒られるか（リスク）：
- ありがちな状況：
- 望ましい対応（OK例）：
- 事前に防ぐチェック項目：

## 章立て（必須）
- セキュリティ／権限
- 運用（ログ・監視・アラート）
- 信頼性（リトライ、冪等性、DLQ、タイムアウト）
- パフォーマンス（コールドスタート、メモリ、VPC）
- デプロイ／変更管理（手動変更、環境差分、ロールバック）
- コスト（無駄な呼び出し、過剰メモリ、ログ垂れ流し）
- 連携別（API Gateway / S3 / EventBridge / SQS / DynamoDB の典型NG）

## 必ず含める代表NG（最低限）
- 実行ロールが強すぎる（AdministratorAccess 等）
- Secrets を環境変数やコードに直書き
- タイムアウト未設定・長すぎ・短すぎ
- リトライ地獄（イベント再試行で同じ処理が複数回）
- 冪等性なしで二重処理
- ログに個人情報・機密情報を出す
- CloudWatch Logs を見ない／相関IDなし
- 本番をコンソールで直接いじる（IaC無視）
- VPC Lambda でNATやDNS周りを理解せず詰む
- 例外握りつぶし・成功扱いにしてしまう

## 注意
- 初学者が「明日から避けられる」レベルの具体性で書く
- 正論だけでなく、現場で実際に問題になる観点（レビュー指摘）を重視する
